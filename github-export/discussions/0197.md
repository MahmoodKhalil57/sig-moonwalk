### Comment 1 — @handrews · 2025-07-15 05:01Z

Notes from the previous meeting taken by @hudlow from our prioritization discussion for 3.3:

-----

Ideas that are concrete and contained enough to consider priorities for 3.3, roughly in a priority order based on appetite/feasibility:

1. @karenetheridge issues surrounding better ways to express things in multiple documents; e.g., multiple documents that each contain paths that can be combined into a single logical OAD that has all the combined paths.
   - Might be interesting to look at the redocly `join` command to see how it works and what assumptions they have to make.
   - @handrews concrete and straightforward — solves a very common pain point; high priority!
2. @jeremyfiel: examples in ~path item~ operation object? @handrews possibly the HTTP message media type? [RFC 9112](https://www.rfc-editor.org/rfc/rfc9112.html#name-enclosing-messages-as-data).
   - See also: https://github.com/OAI/OpenAPI-Specification/discussions/3874
   - Maybe more broadly, this is about correlating request and response examples.
   - @handrews concrete/contained enough to consider higher priority
3. How to model headers more effectively. Headers have structure, but we don't model them well and really only allow treating them as a string.
   - @handrews Background: current tools for modeling headers are inadequate, need to support something akin to media types with mapped HTTP and JSON representations. Note that this is probably a prerequisite to using headers for signatures.
   - for the `Link` header, there's already a media type and a well-defined way to convert between that media type and a JSON structure, so **this is even a possibility for 3.2** — in any case, it sets the precedent for what is needed for structured headers and more specialized structures.
   - See also https://github.com/OAI/sig-moonwalk/discussions/22
   - @handrews First step is to model the generic syntaxes:
     - RFC 8941 defines structured headers, but there's no media type and no trivial mapping to JSON. 
     - RFC 9110, current HTTP semantics defines general header syntax; compatible with structured fields, but closer to historical precedent with some notable exceptions. 
    - Lots of headers with their own bespoke historical formatting rules, some that make sense others that are bizarre (e.g., `Set-Cookie` ... `Cookie`?) — deprecated behavior only helps so much as it may still be used in the real world. 
4. Following on 3,  a clear media type or media range with parameters (for request matching and maybe also response matching) / JSON mapping would be a good 3.3 enhancement
5. @pavelkornev: template paths in security schemes: https://github.com/OAI/sig-moonwalk/discussions/50
   - @handrews Definitely concrete enough to be on the higher side of priorities, but some more use case work needed to determine how to strike the right balance of flexibility vs continuity with what already exists.


Bigger, less scoped ideas that need more refinement before they can be prioritized:

- What are some small, valuable pieces of the ideas in signatures that we could add in 3.3?
  - See also https://github.com/OAI/sig-moonwalk/discussions/16
  - @handrews key original ideas:
    - matching on _more than_ the path template we have today
    - _explicit_ disambiguation of some sort
    - correlating requests and responses
  - @handrews Signatures are fundamentally about matching and correlating, which are two separate things.
  - possible first steps:
    - reframe current behavior in these terms. Consider path templates _as_ signatures, consider the ambiguity and resolution, consider tools now available for correlating requests and responses.
    - Figure out what people are actually asking for here:
      - Gary: would like to use signatures for new version of CDMI, and one of the items that comes up is that there are UUIDs for each type of object/operation, and signatures would be a consistent vehicle to identify the type in "real time." Would be used for path parameters, query parameters, bodies, etc.
      - @hudlow is also interested in using signatures to correlate request and response schemas.
- @hudlow adds: maybe consider modeling more fine-grained IAM systems that don't map well to scopes.
  - @handrews: @sensiblewood is the right person to collaborate with on this, there is definitely some interest, but it's a big, complicated topic.
  - @hudlow also adds this fits into the signature discussion because a motivator for carving up operations into finer-grained signatures can be wanting to describe how authorization differs for requests based on parameters, bodies, etc.
- Broadly: a more formal separation between deployment and shape. (A shape of the API might be hosted on different servers with different endpoints and security restrictions.)
  - Related to overlay, but a different emphasis
- More tools for defining resources are a very hot item (see https://github.com/OAI/sig-moonwalk/discussions/30) but we need to put some effort into honing this into specific ideas.
  - @hudlow: in theory you could have a top level "resources" property that just glued existing things (mainly operations and schemas) together into resource definitions.
- @hudlow still thinks assertions are a relevant topic at this level, but this is controversial, and needs fleshed out.
