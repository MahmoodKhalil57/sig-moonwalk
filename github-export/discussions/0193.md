### Comment 3 — @hudlow · 2025-06-17 16:41Z

Real-time notes:

@handrews in the context of reframing ourselves as the idea lab for 3.3 and beyond, what are the big ideas we should tackle in 3.3, keeping in mind this is a space to talk about features and not the nitty gritty of getting them merged. (If we back ourselves into a corner in 3.x, we can still discuss what something would mean in 4.0.)

Let's tie ourselves to things that people explicitly want or that are coming up/hot in the industry (e.g., meeting the needs that MCP meets). "What problem does moonwalk solve?" We need a focused answer for this.

@hudlow asks "does this mean we should focus on user stories?": @handrews yes...? (Not trying to get pedantic about the format of requirements, but we do want to focus on solving specific problems.)

@jeremyfiel asks about getting feedback from tooling developers; consensus is this really is a big problem — meeting times, chasing people down — people are focused on their short timelines versus the longer timelines of spec evolution. Need to follow up more broadly.

On the topic of implementation difficulties: @handrews was tasks with coming up with a compliance suite, but many things not really able to test — hence DOM and ADA which would empower this effort and make things easier for tooling developers. With 4.0 on the back burner, it's unclear what the future of DOM and ADA are, considering that it's maybe not that likely that 3.x implementation will move to them versus upgrading their own internals to support each next version.

A tool might accept something as valid, but that might not be meaningful if it's unclear what it thinks it _means_. Given an example request or response, if a tool can correctly determine whether the payload is valid and a part of the API and deserialize it into the proper internal representation, this could be another useful approach distinct from the more comprehensive DOM/ADA treatment.

Specific ideas:
- What are some small, valuable pieces of the ideas in signatures that we could add in 3.3?
  - See also https://github.com/OAI/sig-moonwalk/discussions/16 
- How to model headers more effectively. Headers have structure, but we don't model them well and really only allow treating them as a string.
  - See also https://github.com/OAI/sig-moonwalk/discussions/22
  - for the link header, there's already a media type and a well-defined way to convert between that media type and a JSON structure, so this is even a possibility for 3.2
  - RFC 8941 defines structured headers, but there's no media type and no trivial mapping to JSON.
  - RFC 9110, current HTTP semantics defines general header syntax; compatible with structured fields, but closer to historical precedent with some notable exceptions.
  - Lots of headers with their own bespoke historical formatting rules, some that make sense others that are bizarre (e.g., `Set-Cookie` ... `Cookie`?) — deprecated behavior only helps so much as it may still be used in the real world.
- Security related topics like OpenID Connect where some ideas are floating around on significant expansion; definitely an area to explore where there have been lots of topics raised.
  - @hudlow adds: maybe consider modeling more fine-grained IAM systems that don't map well to scopes.
  - @pavelkornev: template paths in security schemas: https://github.com/OAI/sig-moonwalk/discussions/50
- Broadly: a more formal separation between deployment and shape. (A shape of the API might be hosted on different servers with different endpoints and security restrictions.)
  - Related to overlay, but a different emphasis
- @karenetheridge issues surrounding better ways to express things in multiple documents; e.g., multiple documents that each contain paths that can be combined into a single logical OAD that has all the combined paths.
  - Might be interesting to look at the redocly `join` command to see how it works and what assumptions they have to make.
- More tools for defining resources are a very hot item (see https://github.com/OAI/sig-moonwalk/discussions/30) but we need to put some effort into honing this into specific ideas.
  - @hudlow: in theory you could have a top level "resources" property that just glued existing things (mainly operations and schemas) together into resource definitions.
- @jeremyfiel: examples in path item object? @handrews possibly the HTTP message media type? [RFC 9112](https://www.rfc-editor.org/rfc/rfc9112.html#name-enclosing-messages-as-data).
  - See also: https://github.com/OAI/OpenAPI-Specification/discussions/3874
  - Maybe more broadly, this is about correlating request and response examples.
- Webhook stuff:
  - https://github.com/OAI/OpenAPI-Specification/discussions/2600
  - https://github.com/OAI/OpenAPI-Specification/discussions/4230
- @hudlow still thinks assertions are a relevant topic at this level, but this is controversial, and needs fleshed out.
