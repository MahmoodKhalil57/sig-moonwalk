## Body

I just checked [the proposed format](https://github.com/OAI/moonwalk/blob/main/examples/dependentParameters.yaml) for specifying inter-parameter dependencies with JSON Schema (thanks @darrelmiller and @mkistler!) and, although it may support most dependencies found in our research study (see [research paper](https://www.researchgate.net/publication/336816181_A_Catalogue_of_Inter-parameter_Dependencies_in_RESTful_Web_APIs), [Medium post](https://medium.com/isa-group/inter-parameter-dependencies-in-rest-apis-4664e901c124) or [ASC'22 talk](https://youtu.be/fyOlqao2iXM)), I see three main drawbacks:

- **Arithmetic and relational dependencies are not supported**. Or, at least, I don't see any examples. In our experience, arithmetic dependencies are not common at all (e.g., `p1 + p2 < 100`), although relational dependencies are very common (e.g., `p1 < p2`). These are very typical for specifying ranges (e.g., `minPrice` and `maxPrice`).

- **Some dependencies are hard to understand**. When reading a description like the [Tropo dependency](https://github.com/OAI/moonwalk/blob/main/examples/dependentParameters.yaml#L32), _"if type == 'number' then prefix or number must be specified"_, I find it really difficult to understand its corresponding translation to the proposed format:
  ```yaml
  oneOf:
    - not:
      - properties:
          type:
            enum: ['number']
    - required: [prefix]
    - required: [number]
  ```
  What's more, I'm not 100% sure this dependency is correct, since the oneOf keyword forces the use of _just one_ alternative, but using both `prefix` and `number` at the same time would still satisfy the dependency (i.e., that combination is an `anyOf`). In any case, to me, it seems much more natural to write something like `IF type=='number' THEN prefix OR number'`. Plus, it is less error-prone. These two dependency formats are not exclusive. We could support a simple DSL like that (something similar to our [IDL](https://github.com/isa-group/IDL)) that _translates_ into this JSON Schema-compatible format.

- **Some dependencies are too verbose**. Some dependencies like ZeroOrOne or Complex dependencies happen to be too verbose with this format. A clear example is the [ZeroOrOne dependency from YouTube](https://github.com/OAI/moonwalk/blob/main/examples/dependentParameters.yaml#L210): 31 lines for something that could be written as `ZeroOrOne(forContentOwner, forDeveloper, forMine, relatedToVideoId)`. Again, this could also be solved by supporting and intermediary succinct DSL that would translate into this format. In this way, we wouldn't have to reimplement all the validation logic already present in JSON Schema from scratch, but simply the translation `DSL -> this format`.

All in all, as I was arguing on the Slack channel, it's not only about supporting these dependencies, it's about _how_ we support them. Are developers willing to write 31 lines of code to formally express something like _"Specify 0 or 1 of the following parameters: forContentOwner, forDeveloper, forMine, or relatedToVideoId"_? I really think we need to make their (our) lives easier.

### Comment 1 — @handrews · 2022-10-04 16:47Z

@AML14 regarding relational and arithmetic constraints, it is possible (in theory) to express such things in JSON Schema, just not with the standard vocabularies.  From the most recent draft (unchanged from the previous draft or two, I'm pretty sure):

> [7.9. ](https://www.ietf.org/archive/id/draft-bhutton-json-schema-01.html#section-7.9)[Loading Instance Data](https://www.ietf.org/archive/id/draft-bhutton-json-schema-01.html#name-loading-instance-data)
While none of the vocabularies defined as part of this or the associated documents define a keyword which may target and/or load instance data, it is possible that other vocabularies may wish to do so.
>
> Keywords MAY be defined to use JSON Pointers or Relative JSON Pointers to examine parts of an instance outside the current evaluation location.
>
> Keywords that allow adjusting the location using a Relative JSON Pointer SHOULD default to using the current location if a default is desirable.

So if you want to say that the `maximumAmount` parameter, if present, is strictly greater than the `minimumAmount` parameter, you would need to define a `greaterThan` keyword like this:

```YAML
parameterSchema:
  type: object
  required: [minimumAmount]
  properties:
    minimumAmount:
      type: number
    maximumAmount:
      type: number
      greaterThan: 1/minimumAmount  # this is a Relative JSON Pointer, using the instance (not schema) structure
```

A basic relational vocabulary would not be hard to spec out.  I know of at least two JSON Schema implementations that can handle extension vocabularies that use Relative JSON Pointer for instance data access.

### Comment 2 — @handrews · 2022-10-04 16:53Z

(I'm starting separate threads to keep sub-topics distinct)

Regarding difficulty of reading, for _"if type == 'number' then prefix or number must be specified"_, rather than this:

```YAML
oneOf:
  - not:
    - properties:
        type:
          enum: ['number']
  - required: [prefix]
  - required: [number]
```

which I agree is incorrect unless the intention was _"exactly one of prefix or number must be specified"_, I would write:

```YAML
if:
  properties: {type: {const: number}}
then:
  anyOf:
    - required: [prefix]
    - required: [number]
```

For this sort of thing I like writing the `const` property on a single line like that.  It emphasizes that there is one simple thing happening, even though normally I would write that kind of schema across multiple lines.  But once you're used to `properties: {<name>: {const: <value>}}` as an idiom, this becomes very clear and recognizable.

Some of this is just education on good idiomatic use of modern JSON Schema.  The OAS community is still used to draft-04 which was far less expressive than 2020-12.

  > I'm definitely not against a DSL that outputs JSON Schema (or any other schema OAS might support in the future).  It is also not mutually exclusive with additional JSON Schema vocabularies (necessary at least for relational assertion support).
  > 
  > What I would caution against is using the most complex example as the driver.  How may systems have 20-parameter dependencies?  How many people really need to understand dependencies at that level?  Is the API specification document really the place to implement such constraints?
  > 
  > One guiding principle with JSON Schema is that there's a limit to how much validation you should try to do in JSON Schema.  Some people would argue that we're well past that limit, others think we're don't do anywhere near enough to be useful.  JSON Schema annotations exist to allow JSON Schema to provide some essential level of validation, and then attach enough information to allow applications to handle the rest.
  > 
  > For example, `format` is an annotation because even when implementations support it as an assertion, the degree of support varies wildly.  `"format": "email"` in one popular implementation is "validated" by checking to see if it contains an `@` character... and that's all.
  > 
  > So an application that wants to validate email addresses but knows that `"format": "email"` is unreliable can check the annotations and run any field annotated as an email address through additional validation.
  > 
  > So one approach would be to add annotations that express the relational constraints rather than directly validating them.  Since the annotation output format is part of the JSON Schema specification, you can easily write a library that consumes the annotation output and performs the additional validation, and expect it to work across JSON Schema implementations (at least those that support at the "basic" output format).  Currently, the biggest barrier to using annotation collection and the output format is that, as specified, it is likely to consume a lot of memory.  A fix for this [is under discussion](https://github.com/orgs/json-schema-org/discussions/236) and one implementation is trying it out to see if it's feasible.
