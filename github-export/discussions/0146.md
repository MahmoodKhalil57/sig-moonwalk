### Comment 1 — @handrews · 2024-07-29 21:14Z

I'd l like to discuss the data modeling area, specifically looking at:
* what things we are trying to model
    * media types (most notably JSON, XML, and forms)
    * grammars (HTTP [headers](https://github.com/OAI/sig-moonwalk/discussions/108) and potentially some query strings)
    * code (basic structures, OO class hierarchies, generics, etc.)
* where in the process of using an API do models appear?
    * SDK input and output (code-oriented) - this is even more important when supporting RPC styles
    * HTTP input and output (media type and grammar-oriented)
    * how do we relate SDK-level and HTTP-level models?
* how well (or poorly) the 3.x approach fits those things and places?
    * our XML approach only covers some parts of XML
    * our approach to forms is inconsistent (query vs body content) missing features (ordering, repeated keys)
    * grammar-based targets such as HTTP [headers](https://github.com/OAI/sig-moonwalk/discussions/108) are not supported well at all
* what other behaviors have gotten mixed into data modeling in 3.x
    * encoding needs to be distinct from data modeling
    * URI Templates conflate them, which makes sense for URI Templates but not for OAS in general
    * Even within URI Templates, the encoding options are not user-friendly
    * Encoding other things is handled in a variety of confusing ways
* how might we separate those other things to make all features more straightforward
     * Need to figure out overlap between JSON Schema `content*` and other mechanisms
     * Possibly `content*` is for encoding _within_ a format's data, while other mechanisms are for encoding based on the target format

I've been spending a lot of time with OAS 3.x data modeling and encoding, and I think we need a deep re-think of how this works:

* Orthogonal concerns should be orthogonal (as much as is possible)
    * data "shape" - the information needed to lay out a data structure in code / memory / db table / etc. that can accommodate any valid instance
    * non-shape validation constraints - information that tells you whether an instance is valid, beyond whether or not it fits the essential shae
    * target format encoding (URI percent-encoding, punycode for international domains, base64 in cookies, base64url, etc.)
    * there is overlap here, but we can be more clear setting expectations for what code gen, in particular, is and is not expected to handle
* Simple cases should be simple, hard cases should be possible
    * most `form-urlencoded` strings (or `form-data` multipart bodies) can be modeled as JSON Objects (unique keys, unordered)
    * but full support requires ordered name-value pairs with additional metadata
    * mapping from one kind of model to another must be treated as a first-class problem rather than a collection of weird hacks scattered across different Objects and standards
* We should define and [register](https://github.com/OAI/OpenAPI-Specification/discussions/3771) modeling approaches for non-JSON data
    * Extensibility, especially for HTTP headers
    * But also for things like JSON sequences that are easy enough to model if you just agree on an approach
    * Code generation patterns
    * the relationship to JSON Schema vocabularies is unclear, but JSON Schema seems to be re-doing all of that so ¯\\_(ツ)_/¯  — I think it's safer for us to use our own registries to manage things
