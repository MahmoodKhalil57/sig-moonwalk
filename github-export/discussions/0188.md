### Comment 3 — @hudlow · 2025-05-14 18:12Z

Here's another perspective. JSON Schema actually does provide a tool from which we can construct "priorities" and statically require unambiguous behavior _and_ allow every operation to be considered in isolation. It's maybe not elegant or attractive, but it does _work_.

Let's say I have two paths, `/{foo}` and `/{bar}`, and their parameters are given two schemas:

```
Foo:
  type: string
  pattern: X
Bar:
  type: string
  pattern: Y
```

Now I have ambiguity that can't be resolved statically, and might or might not actually result in a runtime collision. How can the schemas solve this?

```
Foo:
  type: string
  pattern: X
Bar:
  type: string
  pattern: Y
  not:
    $ref: '#/Foo'
```

And if it's three schemas?

```
Foo:
  type: string
  pattern: X
Bar:
  type: string
  pattern: Y
  not:
    $ref: '#/Foo'
Baz:
  type: string
  pattern: Z
  not:
    anyOf:
    - $ref: '#/Foo'
    - $ref: '#/Bar'
```

Again, I don't find the solution of forcing authors to use explicit `not` applicators in their schemas so we can statically analyze them to be non-overlapping is very _satisfying_, especially when it still means the best a client can do to make sure it invokes the operation it means to is pre-run the input through the schema and throw an error if it fails (not having any other recourse). But I figure it's worth mentioning this as a way of expressing mutual exclusion.
