### Comment 2 — @hudlow · 2025-04-29 04:21Z

My exercise proved more difficult than I had imagined. I started down the path (heh) of something along these lines:

```typescript
interface Definition {
  pathTree: PathSegment
}

interface PathSegment {
  elements: PathSegmentElement[]
  suffixes: PathSegment[]
  operations: Operation[]
}

type PathSegmentElement = StringLiteral | PathParameter
```

... but immediately there are some pretty grisly problems if you assume this data structure is derived from a list of (pseudo-)RFC-6570 path templates. You get the weirdness of "segment elements" and you have to figure out when two segments are equivalent and can be combined and have their suffixes concatenated.

I still think this is a useful and reasonable ADA construction, but:

1. I do not think it is something we could reasonable back-port to 3.x
2. I think it would inform a pretty different direction for Moonwalk.

On that second point, here are some ideas I think would be helpful:

I think path _"templates"_ get us into trouble because they imply a very specific use — taking the template and some values for its parameters and compiling it into a concrete path. But this is a very narrow use case and actually not very interesting for the ADA?

Rather I think we should consider a different idea, very possible path _"formats"_. Some helpful qualities for formats would be:

1. A segment (between two `/` characters) is either parameterized as a single parameter, or it is a literal. Fundamentally, `/{a}{b}/` leads to so many ambiguities outside of the narrow templating context and it requires delving into the schemas defined for the parameters in order to have a prayer of teasing out how you would _match_ on it. Having to bake JSON Schema into your matcher is pretty undesirable for many reasons. And I tend to doubt you actually lose much functionality of real-world utility.
2. As far as path formats are concerned, parameters are nameless and typeless[^typeless].

Formats would probably look something like `/foo/?/bar/?`. If two operations shared a path format, then[^path-signature] their signatures would have to be differentiated by something _besides_ the path.

As far as static differentiation, you could either consider any potential collision invalid[^collision] or you could have a priority order[^order].

[^typeless]: Okay, this part will probably some people unhappy? Someone will want to have two separate operations for `/foo/{type=uuid}` and `/foo/{type=integer}`. And the "format" term actually suggests a solution to this: `/foo/%s` vs `/foo/%d` — but of course these are actually ambiguous and coming up with a generic way of defining statically-determinable-non-overlapping string formats is something we're trying to avoid. I'll note for completeness though that if one was satisfied by breaking ties with specificity in general (as discussed more below) then you could come up with a specificity order for a few string formats (like integer beats float, float beats string).
[^path-signature]: Regardless of differing names and schemas they defined for the parameters.
[^collision]: Which is nicest from a declarative perspective, and probably my preference since I generally don't like the idea that matching a signature can't be done in isolation.
[^order]: Literal segments beat parameterized segments (`/foo` beats `/?` when matching `/foo`), and literal segments further left beat literal segments further right (`/foo/?` beats `/?/bar` when matching `/foo/bar`).
