### Comment 2 — @earth2marsh · 2025-04-29 17:02Z

Today we walked through Dan's [straw-person for ADA interface for path-template-based signatures](https://github.com/OAI/sig-moonwalk/discussions/185#discussioncomment-12976508). This touched on:
- Considering one use case of what it takes to route incoming traffic to backend services
- That it seems shrewd to have priority as an disambiguator of last resort (assuming that there are always boundaries of ambiguity)
- The idea that you should be able to take the ADA and give you a clear answer on whether it can generate working code or not.
- That ADA should do the level of checking that any tooling should need
- ADA should make it easier for an application that reads an OAD to fail out when it hits something it cannot handle.
- Remember that vendors are used to disregarding parts of the spec that don't apply to them or appear too complicated to implement
- That DOM and ADA _are not requirements_ for tooling. You could very well write a low-level parser that’s not concerned with anything DOM or ADA related. But when you are doing this, we strongly suggest you follow our DOM and ADA roadmap to maintain consistency across tooling. (Though if they want to demonstrate compliance some day, this may be necessary.)
