### Comment 1 — @handrews · 2024-08-05 20:20Z

I'd like to continue discussing the Data Modeling area - see [last week's comment](https://github.com/OAI/sig-moonwalk/discussions/146#discussioncomment-10184006) for what we have discussed to date.

First, we should discuss any questions from last week that folks have thought of in the meantime.

After that, here are some topics I'd like to consider, probably in this order although if someone really wants to change the order that's fine:

* SDK-level modeling
    * What do code generation (OAD-to-code) tools need from data modeling?
    * What do spec generation (code-to-OAD) tools need from data modeling?
* HTTP-level modeling
    * What do HTTP serialization / parsing tools need from data modeling?
    * Where is the boundary between HTTP interaction modeling and data modeling?
    * Is there a distinct target format serialization / parsing component (e. g. URI percent-encoding, HTTP header escaping) and if so, what is the boundary between it and data modeling?
* Static vs Runtime usage
    * Can we classify some constructs as essential to static usage such as data structure / class generation?
    * Can we exclude some constructs from static usage, making them purely runtime (which could include "code generation" in the form of constructor or accessor logic, but not the data structure "shape")
* Mapping between SDK-level and HTTP-level modeling
    * Do we need to be able to use the same model for radically different target formats? (I'll reply to this comment with an example)
    * How, and how well, do Parameter Objects support SDK-level interfaces today?
    * What are our "worst case" scenarios that are the most difficult to support today
        * HTTP headers are essentially unsupported except for the most trivial usage
        * Form data either performs complex RFC6570-ish mapping (`schema`/`style`) or leaves it entirely to the user (`content`)
        * XML support is... partial
    * What are our extensibility needs?

### Comment 2 — @handrews · 2024-08-06 16:57Z

Polymorphism (including `discriminator`):
* Important, widely used
* Optional field would be helpful
* Need to support class hierarchies without `discriminator`
* Need to support shape variants that are not class hierarchies (e.g. alternative web forms for a branch in a form flow)

SDK:
* Want to represent a resource as a class, and end users to work with the resource regardless of read/write/etc. etc.
* We've handled `readOnly`/`writeOnly` in different ways in different releases that solved different subsets of the problem

Gary's use case groups (for thinking about discriminator and other data modeling things):

* client-side
* server-side
* tooling use case
