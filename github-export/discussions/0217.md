# **Moonwalk SIG Update: 2026 Orientation**

## **Status and Leadership**

The Moonwalk SIG is shifting its focus for 2026\. For at least the next six months, the intention is to focus on how OpenAPI intersects with LLMs as a new class of smart client.

Organizationally, **1-2 additional co-hosts are needed,** a collaborator to help organize and host sessions, manage the backlog, and drive consensus. If you are interested in that, please reach out to @earth2marsh via the OAI Slack.

## **Strategic Focus: The LLM-Mediated API Landscape**

As the work orients toward supporting a world where LLMs represent a new category of smart client, the goal is to move beyond the assumption of deterministic, human-coded clients. The focus is a model where APIs surface capabilities to low/no-code users interacting via autonomous or semi-autonomous agents. This shift requires re-evaluating what information an OpenAPI Description (OAD) must contain to be truly "agent-ready."

### **Proposed Roadmap (6-Month Horizon)**

*Note: This roadmap is a draft to promote discussion. The community is invited to discuss and refine these priorities and identify gaps.*

1. **Capability Discovery Phase:** Identifying how to signal *what* a service can do, rather than just *how* to call an endpoint.  
2. **Contract Evolution:** Moving from a purely technical contract to a "social" or "intent" contract that agents can interpret.  
3. **Workflow & Overlay Integration:** Determining how Arazzo (workflows) and Overlays can be used to tailor API surfaces for specific LLM-based journeys.  
4. **Drafting Recommendations:** Developing initial sign-posting standards for capability-based access.

## **Open Questions for Discussion**

The following questions are proposed to orient upcoming sessions. The SIG is invited to discuss these without the immediate pressure of finalizing implementations:

* **Capability Surfacing:** When technical details are delegated to an LLM or orchestration layer, what metadata is required in an OAD to effectively surface a "capability"?  
* **Sign-posting:** How can sign-posting for available capabilities be improved to support on-demand, non-linear user journeys?  
* **The Contract Shift:** How should the "social contract" of an API be defined versus the traditional technical contract?  
* **Organization:** Can `tags` and `tag-kinds` be leveraged to logically group capabilities for agents?  
* **Intersections:** How do Overlays (separating layers of responsibility) and Arazzo (workflows) intersect with agentic access?  
* **Optimization:** Can a service's complete description be simplified or optimized specifically to support LLM-based journeys (e.g., an `llms.txt` equivalent for APIs)?  
* **Discovery:** Is the discovery of these capabilities within or out of scope for the Moonwalk project?

## **Participation**

Interested contributors and potential co-hosts should join the next SIG meeting. Details for the meeting schedule and the project backlog can be found on the [SIG-Moonwalk GitHub repository](https://github.com/OAI/sig-moonwalk).```
