  - **↳ Reply 1 — @handrews · 2024-06-18 14:12Z**
  > I've been thinking about a high-level functional area organization of the spec, and at the moment here's what I have:
  > 
  > * parsing, manipulating, and serializing OpenAPI Descriptions
  > * resolving import/reference IRIs to URLs and retrieving them
  > * modeling data
  > * modeling http interactions (API shapes)
  > * securing an API
  > * deploying an API
  > * documenting an API for human consumption
  > 
  > Note that establishing a base IRI and resolving relative IRI-references to IRIs is in the first bullet point; the second is only about mapping identifiers to locations and retrieving resource representations from locations.  These two bullet points should be implemented as separate components (with many tools outsourcing the second bullet, as they outsource to reference "resolvers"/removers today).  The first bullet point also includes a bundled single-document form.

  - **↳ Reply 2 — @lornajane · 2024-06-18 17:02Z**
  > After today's discussion, I think this is a good structure. How these pieces interact with one another is still open to discussion. My personal reflection is that security and deployments should remain independent of one another.
