### Comment 1 — @hudlow · 2025-05-06 16:03Z

Last week, we discussed challenges in designing a path template/format that has the flexibility users demand, but can be statically assessed for collisions. I don't think it likely we can have our cake and eat it here, but here are some ideas for mitigating this as an issue:

Dealing with collisions:
1. Define operations in a cascade, prioritizing the first match.
2. Define rules of specificity such that a more specific signature match is prioritized over a less specific signature match.
3. Include a priority for an operation in case two paths match and requiring differing priorities when static analysis detects a possible collision.

Supporting different use cases:
1. Support a strict validation mode where any potential path collision is reported as an error.
2. Define explicit syntax for path parameters that can expand to multiple segments.

### Comment 3 — @hudlow · 2025-05-06 21:26Z

To the best of my understanding (please suggest corrections):

| scenario                                            | example 1       | example 2       | 3.x                               |
|-----------------------------------------------------|-----------------|-----------------|-----------------------------------|
| different parameter names                           | `/foo/{bar}`      | `/foo/{baz}`      | illegal                           |
| sub-segment[^sub-segment] differing parameter names | `/foo{bar}`       | `/foo{baz}`       | illegal[^hierarchy]               |
| parameter vs literal                                | `/foo/*`          | `/foo/bar`        | specificity precedence            |
| sub-segment parameter vs literal                    | `/foo*`           | `/foobar`         | specificity precedence            |
| parameter/literal vs parameter/parameter            | `/foo/bar/*`      | `/foo/*/*`        | ?[^partial-parameterization]      |
| bidirectional parameter overlap                     | `/foo/*`          | `/*/bar`          | undefined[^ambiguous]             |
| sub-segment bidirectional parameter overlap         | `/foo*`           | `/*bar`           | undefined                         |
| different parameter schemas                         | `/{type=integer}` | `/{type=boolean}` | inexpressible[^different-schemas] |
| super-segment parameterization                      | `/foo/bar/baz`    | `/**/baz`         | inexpressible                     |

[^hierarchy]: The spec says, "Templated paths with the same hierarchy but different templated names MUST NOT exist as they are identical" and I don't really understand what the word "hierarchy" is doing there and if it's load-bearing.
[^ambiguous]: The spec says, "In case of ambiguous matching, it’s up to the tooling to decide which one to use."
[^sub-segment]: I included the sub-segment examples because I think they _feel_ different even if there's not necessarily a reason or precedent for them to _act_ differently.
[^partial-parameterization]: The spec says, "When matching URLs, concrete (non-templated) paths would be matched before their templated counterparts." This doesn't account for two paths where both are parameterized but one has a superset of the other's parameterization.
[^different-schemas]: Technically, since you can define schemas at the operation level, the same path parameter could have different schemas for different operations, but you certainly can't use a path parameter schema to discriminate between two "signatures" that would otherwise collide.
