  - **↳ Reply 9 — @handrews · 2024-08-24 19:07Z**
  > @kevinswiber yeah I think you're right, we should move docs to generated.  Web forms are another generation use case, and one that is sometimes integrated with docs to provide a "try this" interface.
  > 
  > In fact, I think we could define the client side of the "generic" part as... assuming you have a web form that accepts all necessary input as bits of JSON (because all input, be it parameters, request bodies, etc. is described by JSON Schemas), what code is needed to produce the HTTP request?  This also tells us what fields are needed for this part of the larger set of use cases.
  > 
  > I was thinking about this more yesterday and started slicing up the spec yet another way:  What Objects/fields are needed to define the minimal _interface_ presented to humans/code/uis/whatever vs what fields constitute the _configuration_ necessary to know how to map input that is valid against that interface to HTTP?  This is relatively straightforward on the client side, and less clear on the server side.
  > 
  > This is related to generated vs generic but might be an even more fundamental way to think about it.  When users ask for things like relational constraints across parameters, they're not really asking us to (for example) model HTTP messages, which is an approach we've discussed for Moonwalk.  They're thinking about the _interface_ to an operation (or resource), as a single thing, and there might be relational constraints within this interface.  The fact that the single interface maps to a range of different locations in HTTP is important, but is properly hidden behind that interface.
  > 
  > Generators project the _interface_ into various domains, while generic tools apply the _configuration_ to convert between the interface and HTTP.  Of the various ways we've discussed to carve up how we look at the OAS, this one is feeling like the most important so far.  The seven functional areas are still important, and somewhat orthogonal.  But they're a little more abstract.
  > 
  > For example, I was getting bogged down thinking about data modeling in the abstract.  If we have data models at the interface layer and the HTTP layer, then we need to handle some sort of mapping between arbitrary data models on either side.  But really, HTTP is well-defined.  Media types are well-defined.  It's only the interface-side data model that varies.  If we concentrate "data modeling" on that interface, and treat the HTTP side as a configurable transformation to a known, rather than free, format, then this might start to get more manageable.  We kind-of already do that with some OAS Objects, but not in a consistent, coherent way.
