### Comment 1 — @earth2marsh · 2025-01-28 18:11Z

Per #16, we're planning to re-open the discussions on how we might organize 4.x around signatures in order to be more inclusive of HTTP API designs that may not fit within the opinions of the structure of 3.x. Examples include [multi-path segments](https://github.com/OAI/OpenAPI-Specification/issues/2653), situations where [query parameters may differentiate between operations,](https://github.com/OAI/OpenAPI-Specification/issues/123), and a much [larger variety of HTTP methods](https://github.com/OAI/OpenAPI-Specification/issues/1747) (even potentially including custom methods). See also [this doc](https://docs.google.com/document/d/1OM9MuiuafpiX2R-cNIhvaf7xJH95sXH7JtsMS0erxfQ) from the last time we discussed this.

Extra credit: try to model how such a variety of styles might be represented in YAML!

  > @rafalkrupinski the limiting factor is finding people who are both willing and able to take on such a complex change and work out all of the necessary details, whether that's selecting an existing alternative, building a better JSON Schema-ish thing, and/or figuring out how to get different alternatives to work without exploding the complexity of the burden on tool developers.

  - **↳ Reply 4 — @rafalkrupinski · 2025-01-29 10:42Z**
  > @handrews I don't think additional complexity is avoidable, since OAI want to keep all future versions at least functionally compatible, i.e. there must be a way to translate old OAD to newer versions. No reasonable type description language can fully replace JSON Schema, which is a validation language. So the only way is to add a new language as an alternative and keep JSON Schema for compatibility.
  > 
  > Also making the new type language interoperable (types from new language referring to schemas and vice versa) would add more complexity for tools, while keeping them separate would make it harder for OAD authors harder to transition.
  > 
  > But I guess there's already a discussion about this somewhere on GH....

  - **↳ Reply 5 — @handrews · 2025-01-29 19:10Z**
  > @rafalkrupinski yeah everything you mention has been discussed in the Tuesday meetings, often multiple times.  It still comes down to availability of volunteers with the necessary understanding.  Schemas will get addressed in some way for Moonwalk, but whether that's a deep reworking of data modeling or just a more well-defined boundary so that other people can more easily experiment with alternatives will depend on who is available to figure out any sort of deep reworking.
  > 
  > Our challenge is not a lack of understanding of the problem or of the range of possible soluitons.  It is purely one of available time and people, given the huge number of things that need to be done to get to 4.0.

  - **↳ Reply 6 — @handrews · 2025-01-29 19:10Z**
  > Also, we did 3.1 to improve the situation with schemas by restoring compatibility.  But that meant that we didn't do much (other than webhooks) outside of the Schema Object.  We need to focus on improving the things that haven't been examined since 3.0 as well now.

  >

  - **↳ Reply 8 — @handrews · 2025-01-30 22:40Z**
  > @rafalkrupinski solutions for alternative schemas [have been proposed](https://github.com/OAI/OpenAPI-Specification/blob/main/proposals/2019-03-15-Alternative-Schema.md) in the past[^1] and [added to our extension registry](https://spec.openapis.org/registry/alternative-schema/) with two values.[^2]  However, we have not seen much in the way of implementations of this old (3.0-era) proposal.
  > 
  > So it's really not as simple as just saying "here you can do different things, go for it!"  It turns out people just... don't.  Maybe that was more of an issue of follow-through on our part.  Shortly after that, we decided to make 3.1 fully JSON Schema-compatible, which made one of the two registered "alternative" schema values obsolete.  I think it was originally mostly motivated by people wanting to use plain JSON Schema (which they can now do by default) or use XSD for XML (a small subset of potential users).
  > 
  > Perhaps a new push on this now would be more productive as there are more possible options out there.  For something like this, getting it in the spec is only half the work.  Or maybe even less.
  > 
  > [^1]: although we seem to have lost track of some supplementary documentation that 404s, not sure what happened there
  > [^2]: that link to the "master issue" should go to the proposal document now- that's why the issue is closed, not because we gave up on it
