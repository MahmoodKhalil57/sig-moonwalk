### Comment 2 — @earth2marsh · 2025-03-11 16:57Z

As Darrel said, today we tried to add clarity around the role of ADA, specifically around OpenAPI schemas. Mike K.'s question is a good articulation: "Does ADA represent JSON Schema part of an OAD as anything other than an object? If it doesn't, is that really solving a problem for tool vendors?"

  - **↳ Reply 1 — @jviotti · 2025-03-21 21:17Z**
  > @mkistler I'm a bit late to the party, but just watched the recording today. Some thoughts on the fixed-point precision problem you presented and JSON Schema in general:
  > 
  > - I don't think this is an issue/limitation of JSON Schema, but of JSON itself (or YAML)
  > - The JSON grammar (primarily ECMA 404) specifies very loose description for "numbers" without taking into account encodings. The IETF grammar suggests IEEE 764 double precision encoding for real numbers and 64-bit signed encoding for integers. However, its only guidance
  > - The issue gets even worse due to programming languages like JavaScript where integers are by default floating point types and you can only really trust that representation to be safe within the 53-bit range or so 
  > 
  > All to say that even if JSON Schema (or any other schema language for JSON) would define a "fixed-point precision type" or anything along the lines, it would be problematic and/or non-implementable in many/most JSON parsers out there. For better or worse, we are all stuck with the limitations and limited interoperability guarantees of JSON. 
  > 
  > Many people perceive this as a limitation of JSON Schema (as in "JSON Schema doesn't let me specify this!") but I don't think we could reasonably fix it without only fixing it for some programming languages nor any other schema language on top of JSON data model could truly fix it. There is only so much you can rely on JSON while staying largely interoperable (unless you support something else other than JSON/YAML)
  > 
  > See https://www.learnjsonschema.com/2020-12/validation/type/ for some more details on ECMA/IETF encodings.

  - **↳ Reply 2 — @jviotti · 2025-03-21 21:36Z**
  > I think this is also related to the "problem of modelling with JSON Schema" that @hudlow mentioned and emitting types in programming languages. At least on numbers/strings, I don't think this is a problem with JSON Schema, but a problem with JSON. The type system of JSON (mainly when it comes to i.e. numbers) is very disjoint from what programming languages support and that is the root of most problems, even beyond the fact that JSON Schema is a constraint-driven languages and validation vs data modelling. 
  > 
  > It is about how you map a JSON type (i.e. a loose abstract number) into a programming language that defines a specific encoding, when even that choice is performed by the JSON parser (with a lot of potential liberty as per the JSON grammar) and not by the JSON Schema / OpenAPI tooling. For example, you can have a C# API client sending an HTTP request with JSON payload that includes a specific 64-bit integer, and the Node.js API server may receive a slightly different "integer" (an illusion anyway as integers don't exist in JSON) and the only reasonable fix is to either fix JavaScript or fix JSON, which will not happen.
  > 
  > Even if JSON Schema (or another JSON-based schema language) would extend the types it "supports", it would be an illusion subjected to the limitations and interoperability issues of JSON. Unless OpenAPI breaks from the constraints of JSON itself, I don't see how these problems could be fixed.

  - **↳ Reply 3 — @jviotti · 2025-03-21 21:57Z**
  > On modelling vs validation outside specific types and the challenges of proper code generation using JSON Schema, at lot of it comes down to JSON Schema supporting pretty dynamic ways of defining data structures beyond what most programming languages would be permit. That said, I think that's an attribute of the specific vocabularies defined by JSON Schema right now and not of JSON Schema itself.
  > 
  > For example, it would be interesting to define a set of additional JSON Schema vocabularies more targeted at data modelling that would be simple and avoid anything that popular programming languages would avoid to model (like `$dynamicRef`). If OpenAPI would let authors define their own "dialect" (as @kevinswiber suggested), some people could chose the "simple modelling JSON Schema", trading expressiveness for better codegen or choose the more dynamic JSON Schema, knowing that tooling could refuse to handle some complex schemas, and OpenAPI would one of the two as a sane default while avoiding the definition of a whole new schema language to rule them all?
