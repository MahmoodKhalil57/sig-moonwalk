## Body

As part of the moonwalk TDC meeting of 2024-06-04 there was an acknowledgement that it would be valuable to capture the set of use cases that OpenAPI descriptions need to satisfy.

OpenAPI Tooling Developer Use Cases
- Request matching 
- Operation Description
- Security definition
- Request Construction
- Response Handling
- Compliance
  -  Linting
  - OpenAPI Validation
- Authoring

Application Developer use cases (API consumption and production)
- Payload validation
- Routing
- Request validation
- Response validation
- Metrics collections
- Quota enforcement
- Test generation 
- Compliance
  -  Linting

- Documentation generation
- API Client code generation


This is brainstorm of ideas, do not consider it final or even partially correct. We will continue iterating on this next week.

### Comment 2 — @garymazz · 2024-06-04 18:44Z

Abstract definitions: rough draft
**Modularity:** 
The ability to separate the API definition into elements or components which may also be separates into other elements or components. Modularity may include API definition files, paths, schemas, and all other named object and properties

**Extensibility:**
The ability to augment and extend previously defined objects and properties via overriding existing objects and properties definitions. 

**Composability:**
The ability to define/construct objects and properties from previously defined objects and properties. Constructed objects and properties  can reference  previously defined objects and properties by unique name or other unique identifier.  Objects and properties may define rules permitting and limiting the types or names/identifiers of previously defined objects and properties used with constructed objects' and properties' definition.
