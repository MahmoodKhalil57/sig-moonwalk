  - **↳ Reply 6 — @mattpolzin · 2024-07-23 15:58Z**
  > I can't commit to meeting attendance, but I am the maintainer of the Swift OpenAPIKit project mentioned above and I'd be happy to talk to its goals, design decisions, things that made migration from v3.0 to v3.1 of the spec difficult given aforementioned design decisions, etc.
  > 
  > I'd love to field questions, but I'll start with a brief introduction to the above and then go into just a bit more in the details below the fold.
  > 
  > Swift OpenAPIKit supports decoding OpenAPI documents, creating or modifying them in Swift code, and encoding them. Decoding and encoding are done generically so that both YAML and JSON are supported. OpenAPIKit aims to be flexible but not infinitely so with Swift types that represent OpenAPI and JSON Schema requirements as tightly as possible so that errors and warnings can be generated as part of the process of decoding OpenAPI documents and any OpenAPI written in Swift code is nearly guaranteed to result in a valid reasonable OpenAPI Document.
  > 
  > <details>
  > In Swift, it made sense to me to (a) lean on the language-standard `Codable` framework for encoding/decoding and (b) lean into the value semantics of the language and nice static guarantees offered by the Swift type system given structured value types. The two points are intertwined a bit so you can't speak to the benefits/challenges of one without the other, but more or less (a) is not as interesting in the context of design discussion in my opinion. I'll briefly touch on (b) which definitely differentiates OpenAPIKit from at least some other implementations of the spec.
  > 
  > Designing OpenAPIKit around sum and product types (strongly typed structs and enums in Swift) has familiar benefits to those who have worked in statically typed languages: the compiler can help you out a lot when you're building an OpenAPI document in-code or accessing parts of an OpenAPI document you've parsed. It also impacts the complexity of representing OpenAPI documents in two large ways:
  > 1. It benefits both maintenance of the library and use of documents in-code for OpenAPIKit to represent equivalent semantics in exactly one way, but it is also important to decode and then re-encode an OpenAPI document without changing the encoded representation because OpenAPI authors have their own reasons for representing things in one way or another. This is a constant give-and-take then when deciding how to represent things like JSON Schema `enum` vs. `const` where the the former with exactly one element is equivalent to the latter (just a small example).
  > 2. It can be incredibly useful to create Swift types for OpenAPIKit that naturally limit possible values to the only allowable options per the OpenAPI specification, but it's also easy to accidentally disallow an edge case at which point the type has gone from useful to buggy. Pre-release periods and major versions (follow semantic version standards) have been the way OpenAPIKit deals with this inevitable conflict, but an introduction of a new major version of the specification has massive ramifications -- what type-level decisions that were very helpful for the previous version of the specification become blockers to adopting the new version? This was the biggest battle with adopting v3.1 of the specification by far.
  > 
  > I'm not trying to argue for v4 to be less drastic than v3.1, but I am definitely more than happy to explain more about what actually is challenging for OpenAPIKit to change and why. I know v3.0 to v3.1 was intended to be a relatively easy transition, and on paper it feels that way, but it also ended up meaning an impressively large amount of work given the way I chose to design OpenAPIKit's types!
  > </details>

  > 
  > We'll be focusing more on 3.x once we get 3.0.4 and 3.1.1 out the door.  Those two releases are setting the new baseline.  They are true patch releases with no new requirements, although some of the clarifications may be surprising enough to shift how people support 3.1 a bit.   >

But it's easy enough to find those, aside from weird `nullable` usage that slipped through the cracks of unclear wording until 3.0.3... difficulties with "breaking" `nullable` by clarification in 3.0.3 is what led us to adopt a more "we recommend X but implementations can do Y if they really want to, it's just not interoperable" approach in 3.0.4/3.1.1.  If folks have implemented the "wrong" thing in good faith because the spec was unclear, we don't want to suddenly define those implementations as non-compliant.  We just want to make it clear that the "wrong" behavior won't necessarily be consistent across implementations.
