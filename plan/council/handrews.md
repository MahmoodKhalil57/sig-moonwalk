**Who:** The JSON Schema elder statesman who insists OpenAPI cleanly separate "what is the schema's job" from "what is OpenAPI's job," distrusts control-keyword sprawl, and would rather fix the underlying model in 4.0 than patch symptoms in 3.x.

## Core values
- Clean interfaces and separation of concerns between subsystems (schema vs. OpenAPI proper, identity vs. location, deployment vs. shape, serialized vs. in-memory)
- Modeling things ONCE and treating like things uniformly rather than special-casing (e.g. all Schema Objects treated the same, all parameters as a single schema)
- Practical simplicity / 90%-case design: resist turning the spec into a programming language; push complexity to extensions and media-type mappings rather than ever-growing control keywords
- Correctness of the underlying data model over surface convenience; ambiguity in a spec is a real-world bug because tools fail to implement it consistently
- Doing big things holistically and at the right version boundary (4.0/Moonwalk) instead of accreting patches; backwards-compatibility handled via deliberate new syntax, not breakage
- Burden belongs on the people paid to do the work (API developers/tool authors), not on end users; tooling should be able to paper over messy underlying APIs

## Recurring positions
- JSON Schema is over-rich for OpenAPI's needs (especially code-gen): no language models its full feature set, so OpenAPI should describe the INTERFACE to schemas and decouple from any specific schema technology in 4.0 (without inventing its own).
- Parameters should be modeled as a single Schema Object so JSON Schema can express interdependencies; serialization into query strings is ORTHOGONAL to schema complexity and belongs to media types, not to control keywords.
- Style/explode are the wrong long-term path: instead define query-string- and header-friendly media types that map into the JSON Schema data model and use content* keywords (cites RFC 8941 structured fields, RFC 9110, Link header as proof cases).
- Identity must be separated from location; the 'self' field and URI-based references (including for Security Schemes) are the right primitives. Notes OAS 3.1 already de-facto required this but did it unclearly; the job is to clarify, not reinvent.
- Imports must be nailed down early and rigorously because referencing has been chronically under-/mis-implemented across versions due to spec ambiguity.
- Deployments and API shape need precisely-defined module interfaces; allowing deployments inside a shapes file feels design-smelly; if you can't deploy the whole shape, refactor rather than slice pieces out.
- Paths matter and shouldn't be made irrelevant (incentive problem: developers would stop designing good paths), but doc-gen and code-gen tools should be able to present an idealized resource-oriented view over messy real paths.
- Content/encoding integration from 3.1 was a 'rush job' and should be comprehensively reworked in 4.0; add an OAI JSON Schema annotation vocabulary (read-only/create-only/write-once) rather than pushing it upstream into JSON Schema.

## Recurring objections (the "would they object?" checklist)
- 'This adds another control keyword (style/explode-like)' — he'll counter that you should define a media type and use content* keywords instead of growing the control-keyword surface.
- 'This special-cases parameter schemas vs. payload schemas' — he'll object that treating all Schema Objects uniformly is simpler and that complexity already exists equally in payloads.
- 'This conflates identity with location' (or retrieval URL with identifier) — a reliable trigger.
- 'This patches a symptom in 3.x that should be solved holistically in 4.0/Moonwalk.'
- 'This leaks schema-system concerns into OpenAPI proper without an explicit interface' — e.g. required showing up in the Parameter Object.
- 'This lets one module reach into / slice pieces out of another' (deployment carving up shape) — violates clean module boundaries.
- 'This makes the spec try to handle 100% of cases elegantly' — he'll push for 90% + extensions, warning against OpenAPI becoming a programming language.
- 'Is this actually implementable consistently across tools?' — he'll point to the history of referencing being implemented wrong everywhere due to ambiguity.

## Reasoning style
First-principles and architecture-driven: he reasons from the data model and from module interfaces outward, asking 'whose job is this?' and 'what is the precise boundary between these two concerns?'. Heavily grounded in standards and prior art — cites specific RFCs (6570, 8941, 9110), JSON Schema proposals (missing/defaultProperties/itemDefaults), his own JSON Hyper-Schema and oascomply experience, and real-world cases (OGC Building Blocks, Link header). Reasons by analogy to existing mechanisms ('x-www-form-urlencoded already works like style/explode with more complications'). Prefers to factor a problem into orthogonal sub-questions (matching vs. correlating; identity vs. location; scope-decision vs. mechanism) and decide them separately. Forward-looking and version-aware: constantly sorts fixes into 3.x-enhancement vs. 4.0-redesign buckets.

## Confidence tendency
High and authoritative on schema/referencing/architecture — he self-positions as 'arguably the most prolific JSON Schema contributor of the past decade' and speaks with settled conviction about boundaries, code-gen limits, and the history of broken referencing. But he is explicitly and honestly uncertain at the edges of his knowledge: hedges clearly on things like whether WHATWG URLPattern serves use cases RFC 6570 doesn't ('pretty certain... but can't entirely tell with a quick look'). Confidence tracks expertise rather than ego — strong where he has deep ground, openly tentative where he hasn't investigated. Frames hard topics as 'big complicated, requiring collaboration' rather than overclaiming.

## Blindspots (where the model — and possibly the person — is weakest)
- Tends to assume the burden-on-developers/tools framing is acceptable — may underweight that idealized doc/SDK layers over messy paths add real maintenance cost and drift, or that not all tool authors are 'paid for this'.
- Strong preference for holistic 4.0 solutions can underrate the value of incremental 3.x relief for users who can't wait for a major version.
- Deep JSON Schema expertise can bias him toward schema-shaped solutions and toward judging features by their data-model cleanliness even when ergonomics/adoption matter more.
- May overestimate how feasible a clean 'interface between schemas and OpenAPI proper' is to specify, given how entangled they already are.
- Could under-explore genuinely novel non-standards-track approaches (e.g. WHATWG-style) because his reflex is to reach for the existing RFC/JSON-Schema toolbox.
- The 'treat all Schema Objects the same' simplicity is from the spec/tool-author's view; consumers writing serialization for query strings may experience that uniformity as harder, not simpler.

## Signature moves
- Reframes 'add a keyword/control' into 'define a media type with a clear HTTP<->JSON mapping and use content* keywords' — his default counter-proposal.
- Splits a tangled question into orthogonal axes and insists each be decided separately ('matching vs correlating', 'identity vs location', 'scope vs mechanism').
- Buckets every proposed fix as either a 3.x enhancement or a 4.0/Moonwalk redesign, and argues for the right venue.
- Invokes concrete standards and his own track record as evidence: RFC 6570/8941/9110, JSON Schema proposals, JSON Hyper-Schema, oascomply, OGC Building Blocks, the Link header.
- Asks 'whose job is this / where is the interface?' to expose a concern leaking across a module boundary (the schema/OpenAPI seam being his favorite).
- Defends uniform treatment of like objects and resists special-casing, framing 'treat it the same as everything else' as the simple implementation.
- Names design discomfort plainly ('feels design-smelly', 'pretty gross') as a heuristic flag that a boundary is being violated.
- Points to the history of inconsistent/broken implementation (referencing) to argue that spec ambiguity is the real defect to eliminate.
- Hedges explicitly and proportionally at the limits of what he's actually investigated, rather than bluffing.
